# Implementation Review — 2026-04-24

Cross-subsystem audit of polycli performed by Claude (Opus 4.7) using five parallel review agents. Produced as a Codex handoff after the v0.4.0 release and the subsequent uncommitted work-tree changes (runtime hardening, `health` command, `prompt-runtime.mjs`).

**Review scope:**

1. Architecture boundary — is `polycli-runtime` a framework?
2. Provider-behavior consistency across 8 provider runtimes
3. Host-plugin parity across 4 hosts (polycli / polycli-codex / polycli-copilot / polycli-opencode)
4. Test coverage + timing four-state integrity
5. Release readiness (versions, channels, drift)

**Framing correction for this review:** `packages/polycli-runtime` and the 8 provider adapters exist **by the user's explicit request** to unify the legacy kimi / minimax / qwen / gemini plugins and extend to more providers. Do not audit its existence as a Path B violation. The live constraints that still apply are:

- no shared base class, no inheritance tree, no template-method `ProviderBase`
- no unified event-schema layer that erases provider-specific semantics
- four-state timing (`measured` / `zero` / `missing` / `unsupported`) must stay distinct
- `cold` / `retry` must remain unimplemented (no fakes)

Under that lens, the registry is a flat dispatch table and is fine. What follows is the actionable substance.

---

## TL;DR

- 206/206 tests pass in 62.8s. Timing four-state is correctly preserved. Fixtures are real CLI replays.
- Three release-blocking bugs: (1) `qwen` swallows `subtype:"error"` events, (2) `gemini` uses string length instead of byte length for the stdin threshold, (3) the bundle ships code whose sources are uncommitted — CI cannot reproduce the current bundle.
- Codex host is the only one of four that is missing from `marketplace.json` and from `release:check`. Either wire it in or document the separate distribution path.
- `polycli-runtime`'s "published" state is ambiguous: not `private`, version `1.0.0`, never published to npm, consumed only via bundle injection. Pick one direction in the `package.json`.
- `docs/polycli-v1-public-surface.md` is stale (says `v1` ships no runtime and no provider adapters). Rewrite or mark deprecated — it is no longer a contract.
- Nothing in the audit rejects the evolution direction. The architectural shape is healthy; the follow-ups are substance, not philosophy.

## Release-blocking fixes (required before v0.4.1)

### F1 — `qwen` missing `subtype: "error"` handling

**File / line:** `packages/polycli-runtime/src/qwen.js:163`

**Evidence:** `claude.js:30` treats an event as an error when **either** `is_error === true` **or** `subtype === "error"`, and the streaming path at `claude.js:307` mirrors this. `qwen.js:163` only checks `event?.is_error !== true`, so a qwen stream event that carries `subtype: "error"` but `is_error !== true` is classified as success. CLAUDE.md explicitly requires: "sync/streaming two-path error handling must align".

**Proposed fix shape:**

```js
// qwen.js around line 163 — replace
if (event?.is_error !== true) { ... }
// with
const isErrorEvent = event?.is_error === true || event?.subtype === 'error';
if (!isErrorEvent) { ... }
```

Apply the same adjustment to any qwen streaming loop that inspects result events (mirror the path in `claude.js:307`).

**Test plan:** add a qwen.test.js case that feeds a stream event with `{ subtype: 'error', is_error: false, message: 'x' }` and asserts the runner surfaces the error (does not treat as success). A fixture-level replay with a real captured qwen error would be better, but the synthetic case is the minimum bar.

**Scope guard:** do not refactor the broader qwen error path; the fix is a two-line predicate change. Do not touch other providers in the same commit.

### F2 — `gemini` uses string length instead of byte length for stdin threshold

**File / line:** `packages/polycli-runtime/src/gemini.js:23` (constant at line 10, call site near line 23)

**Evidence:** `claude.js:10` and `kimi.js:9` both define `PROMPT_STDIN_THRESHOLD` in bytes and compare with `Buffer.byteLength(prompt)`. `gemini.js:10` also names the constant `PROMPT_STDIN_THRESHOLD` but the call site compares with `prompt.length`. For ASCII inputs the two are equivalent; for CJK / emoji prompts gemini will switch to stdin earlier than the other providers. This is an inconsistency, not a correctness bug, but it is cheap to remove.

**Proposed fix shape:**

```js
// gemini.js — replace
if (prompt.length > PROMPT_STDIN_THRESHOLD) { ... }
// with
if (Buffer.byteLength(prompt) > PROMPT_STDIN_THRESHOLD) { ... }
```

**Test plan:** add a gemini.test.js assertion that a CJK-heavy prompt whose `Buffer.byteLength` is below the threshold uses argv (matches claude's behavior), to lock the intent.

**Scope guard:** one-line change. No other gemini behavior touched.

### F3 — Uncommitted sources that bundles already depend on

**Files (work-tree, untracked):**
- `plugins/polycli/scripts/lib/prompt-runtime.mjs`
- `plugins/polycli/scripts/tests/prompt-runtime.test.mjs`
- `plugins/polycli/commands/health.md`

**Evidence:** all four `plugins/*/scripts/polycli-companion.bundle.mjs` files show +350 lines in `git diff` and the bundle content references `// plugins/polycli/scripts/lib/prompt-runtime.mjs`. But the source is not in git. Anyone cloning main and running `npm run build:plugins` will regenerate bundles **without** the prompt-runtime logic and diverge silently from what is already tracked.

**Proposed fix shape:** `git add` all three files along with the rest of the work tree, run `npm test` to confirm bundles are reproducible, then commit.

**Test plan:** `npm test` must be green after commit. Manual verification: delete each `plugins/*/scripts/polycli-companion.bundle.mjs`, re-run `npm run build:plugins`, and `git diff` must be empty.

**Scope guard:** do not squash F1, F2, F3 into one mega-commit. F1 / F2 are behavior fixes that deserve their own commit; F3 is a hygiene commit that brings the tree into a reproducible state.

### F5 — Disambiguate the status of `packages/polycli-runtime`

**File:** `packages/polycli-runtime/package.json`

**Evidence:** no `"private": true` flag, `version: "1.0.0"`, no npm publish record (npm registry 404), consumed only via `scripts/build-plugin-bundles.mjs` injection into host plugins. External observers reading the manifest will assume it is a stable public package and may try to depend on it. That assumption is wrong today, and the ambiguity will rot over time.

**Proposed fix shape (pick ONE):**

- **Option A (recommended for v0.4.1):** add `"private": true` to `packages/polycli-runtime/package.json` and to `packages/polycli-utils/package.json` and `packages/polycli-timing/package.json` if they are also bundle-only for now. This makes it impossible to accidentally `npm publish` the package and states the intent in the manifest.
- **Option B:** publish `@bbingz/polycli-runtime@1.0.0` to npm, add an entry to `.claude-plugin/marketplace.json` metadata or `README.md` listing it as a consumable library, and document the exported surface in a new `docs/polycli-runtime-surface.md` modeled on `polycli-v1-public-surface.md`.

Do not leave both options open.

**Test plan (Option A):** `npm publish ./packages/polycli-runtime --dry-run` should now refuse with "cannot publish private package". No other test impact.

**Scope guard:** this is a single-field change if Option A is chosen; do not refactor the bundler or exports map.

### P3 — Cross-host command-map doc

**Why:** the four hosts expose functionally equivalent capabilities through very different surfaces: 10 slash-commands (`polycli`), 1 skill-with-subcommands (`polycli-codex`, `polycli-copilot`), 2 tool functions (`polycli-opencode`). A user moving between hosts has no Rosetta stone.

**Proposed shape:** new file `docs/host-command-map.md` with a 4-column table — capability, Claude Code invocation, Codex invocation, Copilot invocation, OpenCode invocation — for all 10 capabilities (setup / health / ask / rescue / review / adversarial-review / status / result / cancel / timing).

**Scope guard:** documentation only; do not unify the host surfaces themselves.

### P4 — Root README capability matrix (including minimax constraints)

**Why:** `minimax` does not support `sessionResume` (capability returned as `false` from `registry.js`). This constraint is surfaced only in `plugins/polycli-opencode/README.md`. Users of the other three hosts have to infer it at runtime.

**Proposed shape:** add a capabilities table to the root `README.md` with columns: provider, streaming, sessionResume, structuredOutput, supported timing metrics (ttft / gen / tail / tool). Source the data from `RUNTIMES` in `registry.js` so it is accurate.

**Scope guard:** README change only; do not change provider capability declarations.

### P5 — Auth-probe inference regexes need tests and a named contract

**Files:** `gemini.js:11-12`, `qwen.js:12-13`, `kimi.js:10-11`, `opencode.js:177-194`, `pi.js:154-171`

**Why:** five providers infer `loggedIn=true` from transient-error heuristics (regex matching of `timeout`, `429`, `ECONNREFUSED`, etc.). The regexes are hardcoded, undocumented, and will silently regress if upstream CLI error messages drift. CLAUDE.md already guards one facet of this ("gemini's timeout/429 inconclusive path must not degrade to loggedIn=false"), but there is no test locking the regex behavior.

**Proposed shape:** extract the transient-error patterns into named constants per provider, add a small test per provider that feeds each pattern and asserts `loggedIn=true with reason='inconclusive'`. No behavior change; just a named contract.

**Scope guard:** no cross-provider unification of the regex set — the patterns are provider-specific and must stay that way.

---

### P6 — Integration tests still mock child processes

**Files:** `plugins/polycli/scripts/tests/integration.test.mjs`

**Why:** provider main paths are covered by real replay fixtures, but the host-plugin integration tests mock `child_process` / `runCommand`. Agent 4 scored test-confidence at 78%, with the 18% deduction mostly attributed to this.

**Proposed shape:** for at least `/ask` and `/health`, capture real fixtures (one success + one failure per provider) and replay them through the host-plugin tests. Model the capture step on `packages/polycli-runtime/test/fixtures/`.

**Scope guard:** start with one provider (e.g., claude) as a pilot; do not mass-record all 8 in one pass.
