## R6 — Auth-probe regex named contract

### Background

Five provider runtimes infer `loggedIn=true` with a "probe inconclusive" reason when an auth probe fails with a transient-looking error (timeout, 429, `ECONNREFUSED`, etc.). This prevents CLAUDE.md's rule "timeout/429 不要倒退回 loggedIn=false" from being silently regressed.

Today the detection is inline regex literals with no name and no tests:

- `packages/polycli-runtime/src/gemini.js` lines ~11–12
- `packages/polycli-runtime/src/qwen.js` lines ~12–13
- `packages/polycli-runtime/src/kimi.js` lines ~10–11
- `packages/polycli-runtime/src/opencode.js` lines ~177–194
- `packages/polycli-runtime/src/pi.js` lines ~154–171

(Line numbers are approximate; find by searching for `timeout|429|ECONNREFUSED` within the `getXxxAuthStatus` body of each file.)

If any upstream CLI changes its transient-error wording, the probe silently flips to `loggedIn=false` on that condition — hard to notice without synthetic testing.

### Proposed fix

For each of the five providers:

1. Extract the transient-error regex(es) into a module-level named constant at the top of the file:

   ```js
   // Regex patterns for upstream CLI errors that indicate the probe could not
   // determine auth state (network flake, rate limit, transient infra failure).
   // Matching these surfaces loggedIn=true with reason="inconclusive" rather
   // than loggedIn=false, per CLAUDE.md's "do not regress on 429/timeout" rule.
   const TRANSIENT_PROBE_ERROR_PATTERNS = [
     /\btimeout\b/i,
     /\b429\b/,
     /ECONNREFUSED/,
     /ENOTFOUND/,
     // ...keep the exact patterns each provider already has; just name them.
   ];
   ```

2. Replace the inline regex with iteration over the constant. No behavior change.

3. Export the constant from each provider module (`export const TRANSIENT_PROBE_ERROR_PATTERNS = ...`) so tests can assert against it.

### Non-goals

- Do NOT unify the patterns across providers. Each provider's upstream CLI has its own error-text surface; the patterns must stay provider-local.
- Do NOT change the probe's decision logic. Same inputs → same outputs.
- Do NOT touch `claude.js` or `copilot.js`. Those providers use direct auth probes, not inference — unchanged.

### Test plan

For each of the five providers, add to the existing `packages/polycli-runtime/test/<provider>.test.js`:

1. Import the `TRANSIENT_PROBE_ERROR_PATTERNS` constant.
2. Assert the array has at least one entry (the constant is populated).
3. For each pattern, feed a synthetic error string that matches it through the auth-probe path and assert `loggedIn === true` with `reason` containing "inconclusive" (or whatever the existing field name is — preserve existing language).
4. Feed an error string that matches NONE of the patterns (e.g., `"401 Unauthorized: bad token"`) and assert `loggedIn === false` — locks the "transient vs. genuine" distinction.

### Scope guard

- One commit, five file edits (five src, five test). No bundle-level changes.
- No changes to `docs/polycli-v1-public-surface.md`, no CLAUDE.md edits, no README table updates.
- If any provider's current regex is actually wrong (misses a pattern the upstream emits), do NOT fix it in this commit. Record the finding as a CHANGELOG candidate for a follow-up and move on — the purpose of R6 is to lock existing behavior, not extend it.

### Suggested commit message

```
refactor: name auth-probe transient-error regex constants (R6)

Extract inline regex literals in gemini / qwen / kimi / opencode / pi auth
probes into module-level TRANSIENT_PROBE_ERROR_PATTERNS arrays, exported for
testability. Adds one test per provider covering matched-pattern (loggedIn
stays true, probe inconclusive) and genuine-auth-failure paths.
```

## R5 — Host integration test fixture pilot (claude `/ask` + `/health`)

### Background

`plugins/polycli/scripts/tests/integration.test.mjs` currently mocks `child_process` / `spawn` to simulate provider CLIs. Provider main paths already have real captured fixtures in `packages/polycli-runtime/test/fixtures/` — host-side tests do not.

This pilot converts two test cases — claude `/ask` success and claude `/health` success — from child-process mock to real CLI fixture replay, **for claude only**. The other seven providers stay on the current mock path for this commit; a follow-up will fan out.

### Proposed fix

1. Look at `packages/polycli-runtime/test/helpers/fixture-replay.mjs` for the existing replay harness model. Generalize it just enough to be reusable from the host integration tests, OR create a host-side mirror `plugins/polycli/scripts/tests/helpers/fixture-replay.mjs` if the package version is tied to runtime-internal types.

2. Capture two new fixtures by running the real claude CLI once locally:
   - `plugins/polycli/scripts/tests/fixtures/claude/ask-ok.stream.txt` + `.meta.json`
   - `plugins/polycli/scripts/tests/fixtures/claude/health-ok.stream.txt` + `.meta.json`

   Capture shape should match the runtime's fixtures (same `.stream.txt` + `.meta.json` pair with `capturedAt` timestamp, expected response, captured session ID).

3. Replace the two claude mock test cases in `integration.test.mjs` with fixture-replay test cases. Keep the other six providers' mock-based tests untouched for this commit.

### Non-goals

- Do NOT fan out to the other seven providers in this commit. Pilot on claude only.
- Do NOT delete any existing mock-based test until its fixture-replay counterpart is in place.
- Do NOT refactor the integration test file layout beyond the two converted cases.
- Do NOT add a `capture` script that runs provider CLIs in CI. Capture is a one-time manual step per fixture.

### Billing note

Capturing fixtures requires one real `claude` CLI invocation per fixture. Keep prompts trivial (`"Reply with only: OK"`) to minimize cost. Do NOT include the `npm run capture:fixtures` in `release:check` or CI.

### Test plan

1. `npm test` stays green. Test count increases by the number of new assertions (at minimum 2 new replay-based cases replacing 2 mock-based cases; likely stays at 250 or rises slightly).
2. `integration.test.mjs` claude `/ask` and `/health` tests read from the new fixtures, not from mocks.
3. Run `npm test` at least once with the fixtures deleted to confirm the replay harness surfaces the missing fixture clearly (don't commit the deletion — just verify the error path).

### Scope guard

- One commit, minimum file surface: helpers (maybe), two new fixture pairs, modified `integration.test.mjs`.
- If the harness is easier to live in `packages/polycli-runtime/test/helpers/` and re-export from there, do that. If not, host-side mirror is acceptable.
- Leave the other seven providers' mock tests as-is. A future commit will convert them one or two at a time.

### Suggested commit message

```
test: claude integration tests use real CLI fixtures (R5 pilot)

Pilots the host-level fixture-replay migration. /ask and /health claude tests
in plugins/polycli/scripts/tests/integration.test.mjs now read captured CLI
stdout from plugins/polycli/scripts/tests/fixtures/claude/ instead of mocking
child_process. Other seven providers stay on the current mock path; follow-up
commits will convert them one at a time.
```
