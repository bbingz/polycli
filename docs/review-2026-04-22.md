# Implementation Review — 2026-04-22

Cross-subsystem review of polycli performed by Claude (Opus 4.7) using four parallel review agents over ~5,200 LoC of source (not including bundles). Produced as a handoff for Codex to act on selectively.

**Review scope (four subsystems, in parallel):**

1. `packages/polycli-utils/src/*.js` + `packages/polycli-timing/src/*.js`
2. `packages/polycli-runtime/src/{registry,spawn,timing,constants,index}.js`
3. Eight provider adapters: `claude / copilot / opencode / pi / gemini / kimi / qwen / minimax`
4. `plugins/polycli/scripts/polycli-companion.mjs` + `lib/*.mjs`

**Total findings:** ~117. This document extracts the actionable subset.

---

## TL;DR

- Path B architecture is intact. No hidden BaseSession, no framework drift, no shared runtime base class.
- Four-state timing semantics (`measured / zero / missing / unsupported`) are **correctly implemented at the atom level** (`capabilityMetric` / `measuredOrZero`) but **leak at the boundary** in three places:
  - `registry.js` non-streaming path doesn't pass `supportedMetrics` → producer-side violation.
  - `aggregate.js` merges `measured` and `zero` into one percentile input → consumer-side violation.
  - `timing.js` silently produces `zero` on clock skew → observer-side violation.
- The "no tools" rule for `/review` is enforced in prompt text but **only hard-constrained via CLI flags for `kimi` and `qwen`**. The other six providers can still invoke tools.
- **Constraint 9 pattern (transient probe ≠ auth failure)** is implemented correctly in gemini and must be replicated in `opencode / pi / kimi / qwen`.
- **Constraint 13 (session id fall-through)** is broken for every provider: `kimi` calls `resolveSessionId` with `stderr` only; the rest bypass the utility entirely.
- **Six error-message paths fall back to `stdout`** as error text. stream-json stdout carries assistant tokens and may include credentials.

---

## One finding to REJECT (conflicts with project decision)

**Runtime-core agent finding #14:** proposed changing `timing.js:66, 73` from `unsupportedMetric()` to `missingMetric()` for `cold` and `retry`.

**Why reject:** the memory entry `project_cold_retry_unmeasured.md` documents this is deliberate. `unsupported` = "provider has no capability to measure this honestly"; `missing` = "measurable in principle, not captured this run". Upstream CLIs do not expose stable cold-start or retry signals. Downgrading to `missing` would corrupt cross-provider aggregates by implying the metric *should* be measurable for all providers.

The current hardcoded `unsupportedMetric()` is correct. Do not change.

## P0 — Ship blockers (6 items)

Each is a one-to-ten-line fix with semantic or data-integrity impact.

### P0-1 — Non-streaming timing path emits false `unsupported`

- **Location:** `packages/polycli-runtime/src/registry.js:252-259`
- **Issue:** `runProviderPrompt` (sync path) calls `attachPromptTiming` without passing `supportedMetrics`. Default is `{}`, so every `capabilityMetric(null, Boolean({}.ttft))` returns `unsupported`.
- **Impact:** any sync call produces a record claiming the provider has no ttft/gen/tail capability — a false statement that pollutes cross-provider aggregates.
- **Fix:** mirror the streaming path — pass `supportedMetrics: getTimingSupport(provider)`. Metrics then resolve to `missing` (the correct state for "supported but not captured in sync mode").

### P0-2 — Cancel-vs-completion race on background jobs

- **Location:** `plugins/polycli/scripts/polycli-companion.mjs:792-817` + `lib/job-control.mjs:140-180`
- **Issue:** worker's completion write and `cancelJob`'s state write are not mutex-protected. A job cancelled by the user can be overwritten by a late-completing worker, ending up as `completed` with real output.
- **Impact:** `cancel` semantics are unreliable; user sees "cancelled" then later sees output appear.
- **Fix:** funnel both paths through a single `withLockfile` around state + envelope writes, and have the worker CAS-check current state before writing its envelope (if status is `cancelled`, skip the envelope).

### P0-3 — Corrupt state file silently overwritten on next save

- **Location:** `plugins/polycli/scripts/lib/state.mjs:73-88`
- **Issue:** `loadState` returns empty default state on `JSON.parse` failure; next `saveState` atomically overwrites with that empty state. All job history is destroyed with no warning.
- **Impact:** data loss on first truncation / corruption.
- **Fix:** on parse failure, rename the existing file to `state.json.corrupt-<ISOtimestamp>` as a backup, then proceed with the default.

### P0-4 — SIGTERM without SIGKILL escalation + no process-group kill

- **Location:** `packages/polycli-runtime/src/spawn.js:22, 54-58`
- **Issue:** timeout path sends only `SIGTERM`. Any CLI that catches SIGTERM to clean up and hangs (common) causes the Promise to never resolve, stdout to grow unbounded. No `detached + process.kill(-pid, signal)` path even when callers spawn detached.
- **Impact:** hard-to-diagnose hangs and memory growth on timeout.
- **Fix:** after SIGTERM, start a 1–3s grace timer; if `close` has not fired, escalate to `SIGKILL`. When `options.detached === true`, use `process.kill(-child.pid, signal)` to reach child-forked helpers (common in `claude`, `gemini`, `qwen`).

### P0-5 — Session-ID UUID regex rejects v6 / v7 / v8

- **Location:** `packages/polycli-utils/src/session-id.js:2`
- **Issue:** regex hardcodes UUID version nibble as `[1-5]`. Modern CLIs (including Claude Code) emit UUIDv7; the helper silently returns `null`.
- **Impact:** session-resume identity is lost for any provider whose CLI upgraded to v7.
- **Fix:** relax the version nibble to `[0-9a-f]` (or `[1-8]`) while keeping the variant nibble strict.

### P0-6 — Aggregator collapses `measured` and `zero` into one percentile input

- **Location:** `packages/polycli-timing/src/aggregate.js:80-92`
- **Issue:** `finalizeMetric` builds one `values[]` array containing both `measured` ms and `zero` ms, then feeds it to `calculatePercentiles`. A consumer plotting p95 for `tool` across providers sees zeroes from providers that never invoked tools mixed with real tool latencies.
- **Impact:** exactly the "statistical time bomb" MiniMax's red team warned about — silent four-state collapse at the aggregation boundary.
- **Fix:** keep `measuredValues` and `zeroValues` as separate arrays. Compute percentiles over `measuredValues` only, and expose `zeroCount` / `measuredCount` so consumers can pick semantics. Also add a derived `capability: "unsupported" | "supported" | "mixed"` field per metric.

## P1 — High risk, next cycle

### P1-A — Replicate the gemini transient-probe pattern across providers

`gemini.js` correctly distinguishes `GEMINI_TRANSIENT_PROBE_ERROR_RE` (timeout / 429 → `loggedIn=true, detail="auth probe inconclusive"`) from explicit auth-failure text. This pattern must be copied into:

- `opencode.js:141-157`
- `pi.js:110-126`
- `kimi.js:87-104` (also: verify `--max-steps-per-turn 1` still exists in current kimi CLI)
- `qwen.js:217-239` (also: drop the nonexistent `qwen auth status` subcommand — detail-leak risk via agent finding 22)

Otherwise all four providers mis-report `loggedIn=false` on transient upstream failures, regressing the behavior constraint from the 2026-04-22 handoff.

### P1-B — Route every provider's session ID through `resolveSessionId`

Constraint 13 says fall-through order is `stdout → stderr → file`. Current state:

- `kimi.js:141, 189` passes only `{ stderr, priority: ["stderr"] }` — violates on all three counts.
- `claude / copilot / opencode / pi / qwen / gemini` do not call the utility at all; they read from stdout JSON only.

**Fix:** in each adapter, after parsing stdout, call:

```js
const sessionId = resolveSessionId({
  stdout: parsedStdoutSessionId ?? result.stdout,
  stderr: result.stderr,
  fileValue: providerSpecificFilePath ?? null,
  priority: ["stdout", "stderr", "file"],
});
```

### P1-C — Drop generic `event.text` fallback in non-assistant parsers

- `opencode.js:66-67` — accepts any event with `typeof event.text === "string"`.
- `pi.js:56-58` — same.
- `gemini.js:38-47` — accepts any event with `content/text/delta` when `role` absent.

**Impact:** `tool_result`, `system`, or `step_finish` events pollute `response` and extend the ttft/tail window, violating constraint 12.

**Fix:** require either an explicit `role === "assistant"` or a known event type (`text` / `message` / `message_delta` / `result`). Drop the generic branch.

Additionally in `registry.js:207-221`, `isTerminalSummaryEvent` omits `gemini`. Add `gemini` with `event.type === "result"` for symmetry with the other streamed providers.

### P1-D — Copilot & OpenCode must capture `type:"error"` events and real result shapes

- `copilot.js:188-192, 247-251` — only checks `is_error` and `exitCode`. Real copilot CLI also emits `type:"error"` as a standalone event and sets `error.message` on `type:"result"`.
- `opencode.js:119-121, 238-240` — same problem; `resultError = parsed.resultEvent?.error` misses standalone `type:"error"` and non-zero `status`.

**Fix:** capture both `type:"result"` and `type:"error"` into the parser's `resultEvent`, and include `error.message` / `status` / `exitCode` in the failure predicate.

### P1-E — Gemini streaming doesn't validate `hasVisibleText`

- **Location:** `gemini.js:207-214`
- **Issue:** other streamed providers (`claude / copilot / opencode / kimi / qwen`) reject `ok:true, response:""` with `"<provider> produced no visible text"`. Gemini trusts child exit code only.
- **Fix:** add the `hasVisibleText` check (`!response.trim() → ok:false, error:"gemini produced no visible text"`) and also honor `parsed.events.find(e => e.error)`.

### P1-F — Atomic write is not crash-safe

- **Location:** `packages/polycli-utils/src/atomic-save.js:23-28`
- **Issue:** `writeFileAtomic` does not `fsync` the tmp file before rename, and does not `fsync` the parent directory after rename. Classic ext4/APFS rename-truncate data-loss window.
- **Fix:** open with fd, `write`, `fsync(fd)`, close, rename, then open parent dir and `fsync`.

Additionally, `atomic-save.js:61-69`: stale-lock reclaim checks `mtime` but the lock owner never touches the lockfile after creation. A legitimate long-running critical section will be mis-classified as stale. **Fix:** have the lock owner periodically `utimesSync` the lockfile, or write owner PID and check liveness via `process.kill(pid, 0)` before unlinking.

### P1-G — Jobs/<id>.json writes are not lock-protected

- **Location:** `plugins/polycli/scripts/lib/state.mjs:146-188`
- **Issue:** `writeJobFile` / `writeJobConfigFile` use `writeJsonAtomic` without a lock. Worker and parent can race on the same envelope, and this interacts with P0-2.
- **Fix:** gate both writers behind the same `withLockfile` as `state.json`, and include a "write only if current status is still `running`" CAS.

### P1-H — Close six `stdout → error` fallback paths (secret leak risk)

Each of these paths, on non-zero exit, sets the error message to `result.stdout.trim()` or equivalent. stream-json stdout contains assistant tokens and may include credentials in tool output:

- `claude.js:247`
- `copilot.js:201-203`
- `kimi.js:135`
- `opencode.js` (non-zero exit fallback)
- `pi.js:174`
- `qwen.js:286-291` (falls back to `parsed.response` which is legitimate output mis-displayed as error)

**Fix:** never fall back to stdout. Prefer stderr tail. If neither, use a curated diagnostic (e.g., `` `<provider> exited with code ${status}` ``).

### P1-I — `/review` no-tools rule is only hard-constrained for kimi & qwen

- **Location:** `plugins/polycli/scripts/polycli-companion.mjs:507-571`
- **Issue:** prompt text says "Do not use tools" for all providers, but only `kimi` (`--no-thinking`) and `qwen` (`--max-session-turns 1`) get CLI-level constraints. Claude / gemini / copilot / opencode / pi / minimax can still invoke tools; claude in particular defaults to `maxTurns: 10` + `permissionMode: "acceptEdits"`.
- **Impact:** review jobs on those providers may actually scan the repo and run commands, violating the behavior rule.
- **Fix:** per-provider hard constraints in `runtimeOptions.extraArgs`:
  - claude: `--max-turns 1 --disallowed-tools ...`
  - gemini: equivalent approval-mode flag
  - opencode / copilot / pi: equivalent max-steps / no-tools flag
  - minimax: constrain in agent config

## P2 — Medium risk (grouped)

### Parser and edge-case bugs

- `packages/polycli-utils/src/args.js:54-73` — short-option concatenation `-r<uuid>` is silently passed as positional. Real CLIs use this shape.
- `args.js:96-103` — double-quoted regions do not honor `\"` escapes; `splitRawArgumentString('"he said \\"hi\\""')` mis-tokenizes.
- `args.js:121-123` — trailing `\` at end of input produces a literal backslash token (shells would error or line-continue).
- `args.js:34` — `--flag=` (empty value) sets boolean to `true` instead of rejecting.
- `packages/polycli-utils/src/parse-stream-json.js:13-19` — with `allowPrefix:true`, only `{` is recognized as JSON start; arrays and bare JSON values are dropped as `blank`, conflating "empty line" with "non-JSON prose".
- `packages/polycli-utils/src/ndjson.js:13-19` — `readNdjson` swallows every IO error as `[]`. Permission denied vs. no-file-yet are indistinguishable.
- `packages/polycli-timing/src/validate.js:43-82` — validator does not enforce `additionalProperties:false` for `metrics`; extra keys pass validation but fail the schema.
- `packages/polycli-timing/src/percentile.js:14` — accepts `p > 100` and clamps to max; should reject. Also nearest-rank for N=2 returns the lower value as p50.

### Timing correctness

- `packages/polycli-runtime/src/timing.js:68-71` — `Math.max(totalMs - ttftMs, 0)` silently produces `measured:0` when `totalMs < ttftMs` from clock non-monotonicity. This poisons the `zero` state. Switch to `process.hrtime.bigint()` or return `missing` when the difference is negative.
- `packages/polycli-runtime/src/timing.js:13-17` — `measuredOrZero` throws on `ms < 0`. A clock rollback inside `runProviderPrompt` will crash the whole call instead of degrading to `missing`.
- `packages/polycli-runtime/src/registry.js:180-186` — when a provider declared `runtimePersistence:"session"` but the current run failed to produce a `sessionId`, the code downgrades to `ephemeral`. This lies about capability. Keep `session` and annotate the failure in `meta`.

### Process / resource

- `packages/polycli-runtime/src/spawn.js:41-51, 61-71` — `finish` clears the timer but does not detach stdout/stderr data listeners, destroy stdio, or guard the `close` handler against calling `decoder.end()` after settle. Resource leak + post-settle callback surprise.
- `packages/polycli-runtime/src/spawn.js:116-119` — no `drain` awaited on stdin writes; large prompts may be truncated.
- `packages/polycli-runtime/src/spawn.js` — no `AbortSignal` support for user-initiated cancellation.
- `packages/polycli-utils/src/stream.js:7-36` — `buffer` is unbounded; a 1GB no-newline stream OOMs the parent.

### Host plugin

- `plugins/polycli/scripts/polycli-companion.mjs:203-222` — `appendPreview` re-reads the full log file for every event for dedup. O(n²). Keep an in-memory tail instead.
- `polycli-companion.mjs:108-126, 203-222` — dedup between `content_block_delta` (incremental) and `result.result` (full final answer) is based on "last N lines", which fails for interleaved deltas-then-full flows. Track cumulative final text, dedup terminal events by prefix match.
- `polycli-companion.mjs:236-270` — `appendTimingRecord` on the foreground success path throws on validation failure, dropping the user-visible response. Wrap in try/catch and write a stderr warning.
- `polycli-companion.mjs:593-618` — `--timeout-ms` is not NaN-validated; `Number.parseInt("foo", 10) === NaN` → `waitForJob` returns immediately.
- `polycli-companion.mjs:379-388` — if `spawn` fails synchronously (ENOENT on bin), `child.pid` is `undefined`; current code writes `status: "running"` without a pid, blocking `refreshJob` dead-pid detection.
- `plugins/polycli/scripts/lib/job-control.mjs:19-27` — PID liveness check is vulnerable to OS PID reuse on long-running sessions. Trust the envelope-written status instead.
- `plugins/polycli/scripts/lib/review.mjs:58-103` — `auto` scope silently returns empty diff when `HEAD~1` is missing (shallow clone, single-commit repo). Distinguish "no changes" from "git failed".

### Copilot message handling

- `packages/polycli-runtime/src/copilot.js:117-119` — `assistant.message` overwrites `response` on arrival after deltas. Correct if `content` is cumulative; incorrect if CLI ever emits per-turn messages. Add a prefix check before replacing.

## Architecture compliance

- **Path B:** no BaseSession, no shared adapter class, no middleware pipeline. Registry is lookup-plus-timing-decoration; providers are plain objects. ✅
- **Four-state semantics at the atom level:** `capabilityMetric / measuredOrZero / missingMetric / unsupportedMetric` correctly encode the four states, and aggregator counts preserve them. The leaks (P0-1, P0-6, and the timing.js clock-skew case) are all at derivation boundaries, not the core. ✅ with fixes required.
- **`cold` and `retry` deliberately unsupported:** code at `timing.js:66, 73` matches the project decision. Keep as-is. ✅
- **No `shell: true`:** `spawn.js` and `process.js` use `shell: false` by default; args arrays; no injection surface from the utilities themselves. Caller-controlled `options.shell` spread is flagged as a lockdown opportunity (P2). ✅

## Suggested fix order

1. **P0 batch** (6 items, each small, high impact on correctness & data integrity).
2. **P1-A** (transient-probe pattern replication) and **P1-B** (session-id fall-through unification) — both are "apply gemini's / utility's existing pattern to the other providers", low-risk refactors.
3. **P1-H** (close the six stdout-as-error paths) — one curated diagnostic helper, applied in six places.
4. **P1-I** (hard-constrain `/review` for the remaining six providers) — requires understanding each CLI's no-tools flag, so schedule after 2-3.
5. **P1-C, P1-D, P1-E** (parser shape tightening) — ideally backed by real-CLI saved-stdout replay fixtures instead of synthetic ones; address together with the test-coverage gaps.
6. **P1-F, P1-G** (atomic-save fsync and jobs/<id>.json locking) — data-durability batch.
7. **P2 & P3** — schedule against the next release or backlog.
